组织与角色-架构即未来第二三章.md

发表于 2018-03-03   |   分类于 架构   |  

How it works?
作为一个工程师对这两章内容中的观点的解读

角色和责任

一个可扩展性的系统是如何工作的，理清与扩展性相关的技术与执行人员的角色和责任。

  1. 要避免无人负责与多个团队负责同一个系统
    要清楚定义谁是决策者，谁负责为决策提供信息
    对于权力下放的注解可以下放任何权力，但是必须对其结果承担所有责任，好的领导应该是把赞扬留给团队，自己对其结果承担所有的责任。
    ps:书中讲到的CEO，CTO的责任可以了解下： 架构师的职责就是确保系统的设计和架构随着业务的发展而发展。
  2. 不应该有任何一个团队把角色和责任作为无法完成工作的障碍。
    记得曾听过一则FB里的话：所有关于FB的事都是你的事。

    组织的设置

  3. 通过标准化来提高组织的效率
    一个不注重代码，文档，规范和配置标准的制定，发布和应用的组织，研发效率和质量必然低下，生产中出现出现严重问题的风险很大。
    注解：尤其当团队扩大的时候，新人占比提高，缺乏规范导致的问题会出现的更多。
    没有标准和规范，团队会在什么是最佳实践上造成误解。
    团队在业务压力的情况下往往也会背离这些原则进行开发，书中举例的常见借口也是我经常用的：“这个服务对系统无关紧要”或者是”影响不大”。
    如果是这样的话，那为什么要开发这个服务呢？所以平常开发中还是要对自己严格要求。不能牺牲代码质量，这是码农最后的坚守了！
  4. 组织的警告信号
  • 过大的组织
    • 沟通不畅
    • 生产率低下
    • 士气低落
  • 过小的组织
    • 不满意的业务合作伙伴
    • 微观管理的经理
    • 过度劳累的团队成员

上述几点都是书中的描述，也有举例说明。其实在一个团队中研发团队属于过小，而研发+产品+QA 又算一个大组织了，所以这些问题都会出现在同一个团队里。
能想到的解决办法就是通过标准化方式来提高组织效率，但是怎么做才是标准化，这都是需要摸索的，坚持社会主义初级阶段一百年.

书中有列了一个组织团队最佳规模的事项checklist，而之前也有阅读到美团一篇技术文章讲如何保证系统高可用上，也会有个系统稳定的checklist，这个方法其实很通用，毕竟人在处理
问题上还是会有盲目或者考虑不周到的，通过事先列出checklist可以更有调理性的解决问题，对于团队新人也能少走弯路。QA同学在进行系统测试也会有个
思维导图来列举出需要测试的方方面面。这就是方法论。

隔壁团队在实行敏捷看板开发的时候，列举了很多注意点，都是大字黑体加重打印在看板上。最后一条是对事不对人，本章也讲到了不同组织架构，以及冲突的解读（非常精彩）。
组织首先是由各种各样的人构成的，组织的活动不可能避免人的交流。有交流必然会有冲突。冲突也有区别：

  • 认知型冲突
    • 围绕着做什么，为什么要这么做 展开
    • 需要更大的视角和经验
  • 情感型冲突
    • 围绕着怎么做，谁来做
    • 基于角色和自身出发

可以看出“对事不对人”就是要团队成员避免情感型冲突，提倡认知型冲突。
